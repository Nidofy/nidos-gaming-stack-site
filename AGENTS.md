# Nido的游戏小栈 Agent Guide

## 项目定位

本仓库用于长期维护个人独立游戏发布站和配套中文宣传内容。站点中文名称定稿为“Nido的游戏小栈”，英文名称定稿为“Nido's Gaming Stack”。网站首先展示正在开发的 TI 模拟器，未来会继续加入小游戏、模拟器、实验性游戏项目和开发日志。

本项目不是单一游戏的一次性 Landing Page。任何实现都必须面向多作品扩展，优先使用 `data/`、`content/`、MDX 和配置文件驱动页面，不应把 TI 模拟器硬编码为唯一业务对象。

## 推荐技术栈

- Astro + TypeScript + Tailwind CSS + MDX。
- 内容优先使用 Astro Content Collections 或等价的结构化内容层。
- 下载、版本、平台、媒体资源、SEO 元数据和中英文文案应从结构化数据读取。
- 如果仓库尚未初始化，先按 `docs/IMPLEMENTATION_PLAN.md` 执行 Phase 0，不要直接堆页面。

## 代码风格

- 使用 TypeScript，避免隐式 `any`，共享类型放在 `src/types/` 或内容 schema 中。
- 组件保持小而明确：页面负责编排，组件负责展示，数据读取逻辑集中在内容层或服务层。
- 避免复制粘贴式页面。新增游戏、下载项、开发日志、公众号落地页时，应通过新增内容文件或数据项驱动。
- 样式优先使用 Tailwind token 和设计系统变量；避免一次性 magic number 大量散落。
- 不引入大型依赖来解决简单展示问题。新增依赖前应说明用途、替代方案和维护成本。

## 内容风格

- 中文与英文内容要保持信息结构一致：标题、摘要、卖点、平台、状态、下载、媒体、FAQ、开发日志入口等字段应可互相映射。
- 中文可以更适合微信公众号传播：更有故事性、节奏更强、适合分享和转发。
- 英文应偏国际玩家、媒体、开源社区和下载页用户：更直接说明玩法、平台、版本、系统要求、授权和媒体素材。
- 文案避免过度承诺，不写无法验证的性能、兼容性、发布日期或平台支持。
- 所有页面都应明确项目状态，例如 prototype、in development、playable demo、released、archived。

## 设计风格

- 整体可借鉴 apple.com 式产品展示方法：极简、大标题、大图或视频、清晰滚动叙事、强视觉层级、大量留白、明确下载按钮、优秀移动端体验。
- 不复制 Apple 的具体素材、图标、字体授权资源、页面结构或文案。
- 网站第一屏应体现当前重点作品，但导航和内容模型必须支持未来多个作品。
- 视觉资产应优先使用原创截图、录屏、生成图、抽象化视觉或自制品牌素材。
- 不使用未经授权的真实赛事、战队、选手、Dota、Valve、The International 等商标和视觉素材作为本站核心资产。可以使用架空化、原创化、描述性表达，例如“国际赛事风格的电竞模拟器”“幻想战队管理模拟”等。

## 质量要求

- 每次实现用户可见功能前，先确认相关 spec 是否已覆盖；缺失时先补 spec。
- 每个页面都要考虑移动端首屏、键盘导航、图片替代文本、语义化标题层级、深浅色可读性和性能预算。
- 下载相关页面必须区分稳定版、预览版、历史版本，并提供校验、平台、文件大小和发布日期字段。
- SEO、Open Graph、Twitter/X Card、微信分享摘要应基于内容模型生成，不要在多个页面手写重复元数据。
- 任何发布信息必须能追溯到 release 数据源，避免页面文案和下载链接不一致。

## 新增内容的工作方式

### 新增游戏页面

- 在内容集合中新增游戏条目，包含中英文基础字段、slug、状态、平台、媒体资源、下载引用和 SEO 字段。
- 复用通用游戏详情页模板，不为每个游戏复制独立页面结构。
- 若游戏需要特殊展示模块，应作为可选 section 数据或独立组件接入。

### 新增下载页

- 下载信息来自 release 数据，页面只负责展示和筛选。
- 每个 release 至少包含版本号、发布日期、渠道、平台、文件、大小、校验值、变更摘要和关联游戏。
- latest.json 应由 release 数据生成或人工同步维护，并在 release 前检查一致性。

### 新增开发日志

- 使用 MDX 或内容集合管理。
- 每篇日志必须关联语言、游戏、日期、标签和摘要。
- 中英文日志可以不是逐字翻译，但主题、结论和关键链接应一致。

### 新增公众号落地页

- 公众号落地页服务中文传播和转化，不替代正式游戏详情页。
- 页面应包含公众号文章读者需要的快速上下文、下载入口、截图或视频、常见问题和返回主站路径。
- 避免只能在微信内理解的私有表达，确保普通浏览器也能阅读。

## 授权与命名边界

- 不把真实 Dota、Valve、The International、战队、选手、赛事标识或视觉包装作为本站品牌资产或核心页面素材。
- 如需描述灵感来源，使用事实性、低风险、非商标化表达，并避免让用户误以为本站与任何赛事、公司或组织有关联。
- 生成图和自制素材不得模仿受保护商标、队徽、官方界面或赛事包装。

## Codex 工作守则

- 先读本文件和相关 `spec/`，再改代码。
- 不要在未确认的情况下切换包管理器。
- 不要直接实现完整 UI，除非任务明确进入实现阶段。
- 不要自动 git commit。
- 遇到不确定产品决策时，在文档中标记 TODO 或假设，并在最终回复中列出。
